# Agent OS v0.3 Roadmap

**Status:** Public roadmap. Subject to revision based on
real-world feedback after v0.2 (Show HN).

v0.3 lifts agent-os from "framework with structural
anti-coasting" to "framework with measured anti-coasting"
plus the first slice of the Skill System (deferred from v1).

## What ships in v0.3

### 1. Gate-fire counts logged + queryable
Every active-task-gate trigger writes to a structured
event log (JSONL append-only, per the existing event-bus
pattern). New CLI flag:

```
output-clock.py --gate-fires --since 7d
```

Reports: count per agent, per reason, per shift. Lets you
verify the anti-coasting gate is firing on real cases vs
catching only synthetic edge cases.

### 2. Runaway-loop detection numeric proof
Add an LLM-call counter per agent per shift (file-based,
lazy increment via cron-manager hooks — same pattern as
heartbeat). Surface in `output-clock.py --status` next to
the existing BUILDING / WORKING / STALE / IDLE state.

Lets you set a per-shift call ceiling and see how close
each agent is to it before the kill-switch fires.

### 3. Skill System v1 — path-scoped rule loader (first slice)
The full Skill System was deferred from v1 (see
`.claude/rules/architecture.md`). v0.3 ships the most
load-bearing slice: a path-scoped rule loader that lets
you put markdown rule files under `.claude/rules/` and
have them auto-load based on glob patterns in the
frontmatter. Plumbing for full skill packages waits for v0.4.

### 4. External-contributor PR template + flow polish
v0.2 ships `CONTRIBUTING.md` (the docs surface). v0.3
closes the loop on the system-level pieces:

- CODEOWNERS-for-forks pattern (so external contributors'
  PRs route to the right reviewers without manual triage)
- Gemini Code Assist GitHub App scope documentation (so
  forks can opt in to auto-review)
- Ruleset preview UX (visualize the merged ruleset before
  applying — avoids the
  `dismiss_stale_reviews_on_push` ×
  `require_last_push_approval` deadlock we hit during v0.2)

## What v0.3 is NOT

Listing the failure modes we're explicitly designing
against, because every adjacent agent framework has been
pulled toward one of these:

- **Web UI / dashboard.** File-based remains the default.
  CLI-first stays the discoverability surface.
- **LLM-agnostic adapter.** Claude Code is the single
  supported runtime. Simpler is the product.
- **Per-call billing instrumentation.** LLM provider
  dashboards already do this. We won't duplicate.
- **Distributed multi-machine deployment.** One process
  per agent, one shared filesystem, full stop. Multi-host
  is a different architecture, not a v0.3 increment.

## Why these 4 (and not others)

The 4 picks above follow the dunk patterns we expect to
hit on Show HN and from early external contributors:

1. **"You claim anti-coasting but show no numbers"** →
   gate-fire counts + runaway-loop counts answer this
   directly. Both are 1-2 day ports of patterns we
   already use internally; no architectural lift.
2. **"What's deferred from v1?"** → Skill System
   path-scoped loader is the most-frequently-asked
   deferred component (it shows up in our internal use;
   external users will hit the same gap).
3. **"How does an external contributor land their first
   PR?"** → CONTRIBUTING.md (v0.2) is the docs surface;
   v0.3 closes the loop with the actual system-level
   pieces.

NOT picking: meeting-system templates v2 (no demand
evidence yet), output-clock dashboard (UI scope creep
per the "What v0.3 is NOT" list), task-engine retry
semantics (covered structurally by lease expiry today).

## Estimated v0.3 ship

2-3 weeks post-v0.2 (Show HN), dependent on signal
volume from external feedback. v0.3 priorities will
re-shuffle based on what shows up in the first feedback
wave.
